# 过渡到JavaScript/Java

当你写到这里，JSON已经能解决大部分“内容类”需求，但如果你遇到以下情况，就该考虑升级：

- 需要自定义逻辑（新AI、复杂反应、动态产出）；
- 想要修改原版行为（不是简单的字段覆盖）；
- 需要运行时计算、事件监听、UI拓展。

## 两条路线

### JavaScript（脚本）

- 优点：门槛低、无需编译、上手快；
- 缺点：性能一般、调试困难、Rhino限制多。

目录结构：

- `mod.json`
- `scripts/`（入口`main.js`）
- `content/`（仍然可用JSON）

常见注意点：

- 使用`require("path")`导入模块；
- Rhino不完整支持ES6特性；
- 适合小范围逻辑增强。

### Java（主流路线）

- 优点：功能完整、性能好、生态成熟；
- 缺点：配置复杂、学习成本高。

目录结构：

- `src/`（Java源代码）
- `assets/`（内容/贴图/语言等）
- `gradle`与构建脚本

适合做大型模组、复杂系统与UI。

## 过渡建议

- 先用JSON把内容“搭出来”；
- 用JS补逻辑；
- 复杂需求再迁移到Java/Kotlin。

## 结语

JSON是入口，Java是天花板。能写到这里，你已经具备读源码、找字段、组合系统的能力了。下一步就是选择一条路线继续深入。
